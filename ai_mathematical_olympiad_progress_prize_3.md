# コンペ概要

<https://www.kaggle.com/competitions/ai-mathematical-olympiad-progress-prize-3>

## コンペの目的

本コンペティションは、**オープンソースのLLMの数学推論能力を、国際数学オリンピック（IMO）レベルまで引き上げる** ことを目的としています。

与えられるのは、代数・組合せ論・幾何・整数論にまたがる、完全オリジナルの数学問題です。これらを解いて、**0〜99999の整数**で答えます。

XTX Markets が主催する AIMO Prize と Kaggle の共同運営で、賞金総額は約 **$2.2M** という非常に大規模なコンペです。

### コンペティションの背景

AIMO（AI Mathematical Olympiad）は今回で3回目です。

- **AIMO1（2024）**: 優勝 Project Numina（NuminaMath 7B、Private 29/50）
- **AIMO2（2025）**: 優勝 NVIDIA NemoSkills（Qwen2.5-14B + TIR + GenSelect、Private 34/50）
- **AIMO3（2025-2026）**: 「クローズドソースとオープンソースのギャップを縮める」がテーマ

数学オリンピックレベルの問題をAIに解かせるという取り組みは、AIの「推論能力」を測る最前線のベンチマークになっています。アドバイザリーコミッティにはフィールズ賞受賞者の Terence Tao 氏も名を連ねています。

## AIMO2からの主な変更点

主催者の Simon Frieder 氏が Welcome 投稿で明示していた、前作からの主な変更点です。

| 項目 | AIMO2 | **AIMO3** |
|-----|-------|----------|
| 回答範囲 | 0–999（3桁） | **0–99999（5桁）** |
| 評価指標 | Unnormalized Accuracy | **Penalized Accuracy**（2回実行して両方正解で満点） |
| GPU | H100（5時間） | H100（5時間、一部 B200 提供） |
| 時間配分 | 15分以内にサーバー応答必須 | **自由**（1問に最大4時間59分使ってOK） |
| チームサイズ | 5名 | 8名（途中で20名へ拡大） |
| ホワイトリスト | 必要 | **不要**（ライセンス適合は自己責任） |
| 追加リソース | なし | **128 H100 GPU + Tinker クレジット**（申請制、ファインチューニング用） |
| 問題難易度 | National Olympiad 級 | **National Olympiad 〜 IMO 級** |

特に重要なのが **回答範囲の拡大（3桁→5桁）** と **評価指標の変更（Penalized Accuracy）** です。
3桁だと「当てずっぽう（guess）」でも一定の確率で当たってしまいますが、5桁になると実質的に guess は不可能になり、「ちゃんと解く」必要が出てきます。

## データセット

### 問題セットの構成

```
Reference Set:  10問  (解答付き、開発用)
Public Test:    50問  (Public LB)
Private Test:   50問  (Private LB, 2回採点)
合計: 110問 (全て新規作成・LaTeX形式)
```

- 全問が **完全オリジナル**（数学者がレビュー済み）で、学習データの汚染（train-test contamination）リスクがゼロ
- 幾何問題も画像ではなく **完全に文章化**（LLMの図形理解は問わない設計）

### 実行環境の制約

- **インターネット不可**、提出は Kaggle Notebook
- H100 1枚（80GB）、提出時は **5時間ハード制限**
- 外部データ・公開pretrainedモデルは利用可
- **モデルライセンス制限**: 商用利用可能なもの（Apache 2.0 / MIT / BSD など）

## 評価指標

### Penalized Accuracy（2回採点）

今回のコンペで最も特徴的なのが、この評価指標です。

Privateリーダーボードでは、各問題が **同じノートブックで2回実行** されます（固定 random seed・固定順序）。

```
両方正解 → 1.0
片方だけ → 0.5
両方不正解 → 0.0
合計最大 = 50.0 点 (50問)
```

**意味するところ**:
- 多数決でたまたま当たる「lucky run」は罰せられる
- 安定して同じ答えを出せる方が偉い
- 高温度サンプリングは「片方正解」になりやすく、ペナルティを受けやすい（と当初は考えられていた）

前回のコンペで Private で沈んだ経験があったので、この「2回採点で安定するか」は今回かなり意識しました。

## 使用できたモデルの参考スコア

主催者が Welcome 投稿で出した、Reference Set（10問）でのモデル参考スコアです。

```
Qwen3-32B-Thinking-2507:  ~4/10
gpt-oss-20b:               ~3/10
gpt-oss-120b:              ~6/10
DeepSeek-V3:               ~7/10
Claude / GPT (closed):    8-9/10
```

OpenAI が公開した **gpt-oss-120b**（Apache 2.0ライセンス）が、商用利用可能なオープンモデルの中では最強クラスで、結果的にほぼ全員がこれを使うことになりました。

なお、上の「6/10」は **1回試行** の参考値です。コミュニティの実測（ShelterW氏）では、**gpt-oss-120b を15回試行して多数決すると 9/10** まで上がりました。「1回だと6点でも、何回も解かせて多数決すると9点」という self-consistency（多数決）の効きの大きさが、このモデルが選ばれた決め手でした。

また主催者は、Public と Private の難易度がほぼ同じであることを公式データで示しています（強いモデルの pass@20 が Public 47.8 / Private 45.5、pass@100 では両セット合わせて未解は2問のみ）。「Private が極端に難しいのでは」という序盤の懸念は否定されました。この **pass@20 ≈ 45.5** という数字は、後述の「多数決で取りこぼす分（Verifierで回収できる伸びしろ）」の議論で重要になります。

# 上位解法

## 全員が使った「定型アーキテクチャ」

AIMO3 では、序盤こそ Qwen 系など色々なモデルが試されましたが、**gpt-oss-120b の登場以降、Public 40〜44点帯の解法は驚くほど似た構成に収束** しました。

```
共通アーキテクチャ:
  gpt-oss-120b (FP8 KV cache)
  + vLLM (single GPU, prefix caching)
  + Jupyter Kernel Pool 16並列 (Python実行を並列化)
  + harmony format (gpt-oss専用のchat template)
  + ReasoningEffort.HIGH
  + 8 attempts/problem (Self-consistency)
  + エントロピー重み付き投票 (weight = 1/entropy)
  + early_stop (4/8 一致で打ち切り)
  + 動的時間配分 (300〜900秒/問題)
```

この「定型アーキテクチャ」をコピーするだけで Public 40点前後は取れる状態で、そこから先は **温度・min_p・タイムアウト・プロンプト** といった微調整と **多少の運**（再実行で ±2〜3点ブレる）の戦いになっていました。
この**定型アーキテクチャの実質上限は Public 44/50**でした。

ただし後で詳しく書きますが、**この定型に「Verifier（検証）や自己批評」を足した公開ノートは、実際には Public 45〜46 を出していました**（コンペ終了後に Code タブを Public Score 順に並べて気づきました）。「44が公開の天井」というのは、実は定型テンプレに限った話だったのです。

## 上位入賞者のWrite-up

コンペ終了後、Kaggleの `writeups` セクションに各参加者の解法が公開されました。
これらの内容を読んで、一番印象的だったのは、**上位陣でさえファインチューニングはしておらず、ほぼ全員が gpt-oss-120b の「定型アーキテクチャ」をベースに、プロンプトと投票方法を磨き込んでいた** という点でした。

### 2位: Vote-dominant Ensembling + Sharper System Prompt（varianceofx 氏）

公開ノートブック（Parthenos氏／Andreas氏のもの）をベースに、**7つの的を絞った改良**を加えたという、再現性の高い解法です。ファインチューニングはなし、純粋な推論最適化です。特に効いたものを挙げます。

**改良1: システムプロンプトに「答えの範囲」を明記（最も効果大）**

AIMO3 はしばしば「答えを 10^5 で割った余り（下5桁）」を求めますが、モデルは問題を正しく解いた上で **割る前の巨大な整数をboxしてしまい0点になる** ことが頻発したそうです。そこで以下を明記：

> 最終的な答えは [0, 99999] の非負整数でなければならない。もし答えが 99999 を超えるなら、それは問題が「下5桁」や「mod 10^5」を求めている。boxする前に問題文を読み直すこと。

> 「この一段落の追加だけで、本来正解だったのに0点になっていた解答群がまるごと救済された。モデルは解き方は分かっていて、答えの形式を思い出させる必要があっただけだった」

**改良3: Tail-Windowed Entropy（末尾256トークンのエントロピー）**

確信度をエントロピーで測る際、ベースは全トークンの平均を使っていましたが、**最後の256トークンだけ**で測るように変更。「序盤の探索は本質的に高エントロピーで、最終答えへの確信度を表さない。答えをboxする末尾こそがクリーンな信号」という理屈です。

**改良5: Vote-dominant な投票（最も特徴的）**

ベースは純粋なエントロピー重み（`weight = 1/entropy`）で、**1つの超低エントロピーな外れ値が 5対1 の多数決を覆せてしまう** 弱点がありました。これを票数優先に変更：

```python
vote_share = votes / total_votes
confidence = 1.0 / max(median_entropy, 1e-9)
consensus  = 1.0 if (votes == max_votes and confidence > 1.0) else 0.0
score = vote_share * 2.0 + confidence * 0.3 + consensus * 0.5
```

票数（重み2.0）が支配的で、エントロピーはタイブレーク（0.3）に格下げ。「自信はあるが間違っている1試行」に投票結果を乗っ取られにくくしています。

その他、回答抽出を`\boxed{}`のみに限定（散文の "final answer is N" を除去）、中央値エントロピーで外れ値に頑健化、「2位が残り全部得票しても逆転不可能なら早期停止」といった改良が入っています。

### 3位: Avoid Frequent Errors プロンプト（SKobayak 氏）

ZaynYu氏の公開ベースラインがベース。gpt-oss-120b をそのまま使い、8並列＋多数決。核心は **データドリブンなプロンプト設計** でした。

1. **誤答ログを420件収集**: リファレンス10問×32回 + 外部の難問50問×2回
2. **gpt-oss-120bより賢いLLMで誤答を分類**（ただし高度すぎる解法を提案されないよう、gpt-oss-120bの正解例も渡して「実行可能な解法パターン」に絞る、という工夫）
3. 観察された誤答パターンから、プロンプトに以下を組み込み：
   - **Pythonで検証させる**（正解時によく見られたパターン）
   - **制約を明示的に抽出・チェックさせる**（制約見落としの誤答が多かった）
   - **探索範囲を絞らせる**（巨大ループで時間切れする誤答が多かった）

特に印象的だったのが **過学習を避ける工夫** です。公開LBやリファレンスに過剰適合しないよう、わざと**コンペ外の50問**で検証していました。「公開ノートが40点に達した状況では、50問のうち『常に解ける35問・常に解けない5問・ブレる10問』に分かれているはず。LBスコアだけ見ると、その不安定な10問に過学習してしまう」という洞察です。

### Nitarach 論文: Model Capability Dominates（Natapong Nitarach 氏, KMITL）

タイ KMITL の Natapong Nitarach 氏は、**"Every Improvement Failed"** というWrite-upを、同時に arXiv 論文（[2603.27844](https://arxiv.org/abs/2603.27844)「Model Capability Dominates: Inference-Time Optimization Lessons from AIMO 3」）として公開していました。

サブタイトルの *「25 submissions, $300+ in GRPO, 15 offline branches - and a one-sentence prompt still won（25回提出、GRPOに$300、15のオフライン分岐 — それでも勝ったのは一文のプロンプトだった）」* がすべてを物語っています。

主な発見：

| 検証した仮説 | 結果 |
|------|------|
| プロンプトを多様化して試行のエラーを脱相関できる | **23以上の実験すべて失敗** |
| 温度を下げてプロンプト多様化に賭ける | T=0.5 + 多様化 = 36/50（T=0.5単独 38/50 より悪化） |

- **モデルキャパシティの差が支配的**: gpt-oss-120b（39.3点）vs gpt-oss-20b（31.0点）の8点差が、プロンプト最適化の振れ幅（±2点）の4倍
- **温度 T=1.0 で十分にエラーが脱相関する**（試行間のエラー相関 ρ̂ ≈ −0.122 で、すでにほぼ無相関）
- **pass@20 ≈ 45.5 vs majority-vote 42** の差（約3.5点）は、「投票が正解を取りこぼしている分（Selection Loss）」で、**Verifierでしか回収できない**
- 同一設定で21回実行してもスコアは σ=1.7 でブレる（**Lottery dynamics**）

### その他の参考になったWrite-up

- **210位「Fighting Against Variance」**: logprob（確信度）は数学的正しさとほぼ無相関という「logprobの幻想」を指摘し、**Summarizer-Judgerワークフロー**（3つのベース解答 → 要約ノード → ツール付き5つのJudgerノードで検証）を実装。後述する「Verifier」の具体例です。Eagle3を使った投機的デコーディングで約40%高速化していました。
- **359位「Entropy-Weighted Self-Consistency」**: Public best 44/50。学術論文の体裁で、T=1.0が最適・8試行が16/32試行より良い・フラットな時間配分が幾何減衰より良い・SFT/GRPOは効かなかった、と網羅的にアブレーションした良レポート。
- **37位「Prompt to shorten thinking」**: 問題順が固定なので、**1問あたりの思考を短く保つ**ことを重視。「指定しないと gpt-oss は別解や別証明を延々と探し続ける」ため、検証を誤答検出ループに特化させたとのこと。
- **1219位「a collection of delightful side quests」**: AIMO2優勝経験のあるチーム。今回は時間が取れず、締切後リリースのNemotron Cascade-2を**層ごとの蒸留で「ソフトクローン」してルール内で使う**（主催者に合法と確認）という離れ業。Private 41。

## 上位陣から見えた教訓

Write-upを読み込んだ結果、当初の私の理解はかなり間違っていたと分かりました。

| 当初の想定 | Write-upから分かった実際 |
|---------|------|
| 「上位はファインチューニング（FT）が必須」 | **2位/3位/359位ともFTを明確に否定**。FTは効かない or 効果が出なかった |
| 「プロンプトは微チューニング（±2点）」 | **プロンプトの磨き込みが2位/3位の主軸**（特に2位の「答えの範囲明記」が最大の改善） |
| 「Private用に低温度サンプリング」 | **T=1.0で十分**（Nitarach論文・359位・私のPri=Pubで一致） |
| 「マルチモデル合議で底上げ」 | gpt-oss-120bが突出しており、他モデルは弱く価値が薄い |
| 「投票はエントロピー重みが洗練されている」 | **logprob/エントロピーは正しさと相関薄い**（2位・210位が指摘）。票数を主、エントロピーを従に |

ポイントは「**プロンプト多様化（複数戦略を混ぜる）は無効**」（Nitarach）だが「**単一プロンプトの磨き込み＋誤答分析は有効**」（2位・3位）という、一見矛盾する2つが両立することです。
そして上位を分けたのは、FTのような重い武器ではなく、**「答えの形式を明記する」「誤答パターンを分析してプロンプトに反映する」「投票で票数を重視する」** といった、地味だが的を絞った推論時の工夫でした。

## 解法

### 1. モデル選定（gpt-oss-120b）

商用利用可能（Apache 2.0）なオープンモデルの中で最強クラスの **gpt-oss-120b** を選びました。
このモデル選定の時点で、Public 35点以上はほぼ約束されたようなものでした。逆に言うと、ここで他のモデルを選ぶ理由はほぼ無いという状況でした。

### 2. Tool-Integrated Reasoning（TIR）

LLMに考えさせるだけでなく、**Python（sympy, numpy, mpmath）を実行させて計算ミスを排除** する手法です。
これが無いと、どんなに賢いモデルでも数値計算でミスをして20点も届きません。AIMO3でスコアを伸ばす上で最も重要な技術でした。

```
LLM「この方程式を解くと...」
   ↓
Python実行: sympy.solve(...) → 正確な値
   ↓
LLM「計算結果は42なので、答えは \boxed{42}」
```

### 3. 16並列のJupyter Kernel Pool

1問につき8回の試行（attempts）を並列に走らせ、各試行の中でPythonを何度も実行します。
このPython実行を高速にするため、**Jupyter Kernelを16個立ち上げてプールしておく** 仕組みを使いました。

サブプロセスを毎回起動するとimportだけで数百ミリ秒かかりますが、Jupyter Kernelならライブラリをプリロードしておけるので、ターン間で1〜2秒短縮できます。8試行 × 16コードブロック × 50問だと、これが大きく効いてきます。

### 4. ストリーミング推論と早期停止

トークンをストリーミングで受け取り、`\boxed{...}`（最終回答の形式）が出た瞬間にストリームを打ち切ります。
答えが出た後の余計なトークン生成を止めることで、大幅に時間を節約できます。

### 5. エントロピー重み付き投票

8試行の答えを単純な多数決ではなく、**各試行の「確信度」で重み付け** して投票します。

```python
# logprobsからシャノンエントロピーを計算
# エントロピーが低い（=確信が高い）試行ほど票を重くする
weight = 1.0 / max(entropy, 1e-9)
answer_weights[answer] += weight
# 最大weightの答えを採用
final = max(answer_weights, key=answer_weights.get)
```

### 6. 動的時間配分

5時間という制約をフルに使うため、残り問題数と残り時間から、1問あたりの時間予算を動的に決めます。

```python
time_left = notebook_limit - elapsed
reserved = (problems_remaining - 1) * base_problem_timeout  # 300sec
budget = clamp(time_left - reserved, base=300, high=900)
deadline = now + budget
```

余裕がある間は1問に最大15分使い、終盤に近づくと5分まで圧縮する、という設計です。

### 7. Retryメカニズム（独自要素）

数少ない自分独自の追加要素です。
8試行すべてが回答を出せなかった（出力が壊れた、ターン上限到達、`\boxed{}`が出ない等）時の保険として、**temperature=0.7 に下げて4試行を追加** する仕組みを入れました。

```python
if not valid_answers:  # 8試行全部がNone
    if time_remaining > 120:  # 残り2分以上
        retry_attempts = 4
        retry_temperature = 0.7  # より決定的に
        # ... 4試行を追加実行
```

ただし、これは「性能の天井を上げる」ものではなく「全滅を救済する延命策」で、上位を狙うための差別化要素にはなりませんでした。

### ファインチューニングをしなかった

主催者が **128 H100 GPU + Tinker クレジット** を申請制で提供していたのに、申請しませんでした。
コミュニティからは CrystalMath（2,129問）や PolyMath といったSFT用データも共有されていたので、これらで gpt-oss-120b を追加学習すれば底上げできた可能性があります。

ただ、これは上位入賞者のWrite-upを見る限り **必須ではなかった**（2位/3位はFTなしで上位達成）ので、優先度は当初考えていたより低かったかもしれません。

### Verifier（検証）ステップなし

8試行の投票だけで答えを決めており、最後に「この答え合ってる?」とLLMに問い直す工程がありませんでした。

Nitarach論文によれば、gpt-oss-120bの **pass@20 ≈ 45.5**（20回試行のうち1回は正解する）に対し、多数決のスコアは42止まりです。この約3.5点の差（投票が正解を取りこぼす分）は、**Verifierでしか回収できない** とされています。ここが一番もったいなかった点です。

```python
# こういうVerifierを8試行の後に挟むべきだった
def verify(problem, answer):
    prompt = f"Problem: {problem}\nProposed answer: {answer}\nIs it correct? Reply CORRECT or WRONG only."
    out = client.completions.create(prompt=prompt, temperature=0.0, max_tokens=5)
    return 'CORRECT' in out.choices[0].text
```

### プロンプトを磨き込まなかった

私のプロンプトは公開44とほぼ同じ「5段階問題解決プロンプト」（UNDERSTAND→EXPLORE→PLAN→EXECUTE→VERIFY）のままでした。

上位2位/3位は、**単一プロンプトの磨き込み（Sharper System Prompt）** や **誤答パターンの分析からの注意喚起追加（Avoid Frequent Errors）** を主軸にしていました。
特に2位の人が「最も効いた」と言っていた **「答えが99999を超えたら mod を取り忘れているので問題文を読み直せ」** という一段落は、私のプロンプトには無く、言われてみれば「正しく解けたのに0点」という最ももったいない失点を防ぐ、極めてコスパの良い改善でした。
「プロンプトはどうせ微差」と思い込んで深掘りしなかったのが、最大の反省点です。

#### Harmonyフォーマット

gpt-oss 系のモデルは、**harmony format** という独自のchat templateを使う必要があります。
メッセージが複数のチャンネルに分かれているのが特徴です。

- `analysis` channel: 内部推論（chain-of-thought）
- `commentary` channel: ツール呼び出し前の前置き
- `final` channel: ユーザーへの最終回答（`\boxed{N}`）

ツール呼び出しは `recipient='python'` のメッセージとして返ってくるので、それをパースしてJupyter Kernelに渡し、結果を会話履歴に戻す、という流れになります。
専用ライブラリ `openai_harmony` を使ってメッセージを組み立てます。

```python
from openai_harmony import SystemContent, ReasoningEffort

system = (SystemContent.new()
    .with_model_identity(system_prompt)
    .with_reasoning_effort(ReasoningEffort.HIGH)  # 思考を長く取る
    .with_tools(tool_config))
```

### 最終スコア

定型アーキテクチャの実質上限（44/50）に近いところまで来られましたが、メダルには届きませんでした。

そして振り返りで一番の学びだったのが、**Code タブを Public Score 順に並べると、公開ノートの実際の上限は 44 ではなく 46 だった**ことです。

| Public Score | 公開ノートブック | 手法 |
|---|---|---|
| **46** | critique-rlm-like | 定型 + **答えを批評して解き直す自己修正ループ** + 共有DBで試行間の合意 |
| **45** | aimo-agentic-solver | **マルチエージェント**（要約 → ツール付き検証エージェント(TRUST/VERIFY/REJECT) → 司令塔が選択） |
| 44 | 44-50（votes 2958）, ans-verifys, with-tools | 定型アーキテクチャ |
| 43 | weighted-entropy, bayesian, **gpt-oss-20b-finetuned**（唯一のFT系だが43止まり） | |

つまり **44→46 を実現していたのは、まさに私が「足りなかった」と挙げた Verifier（検証）や自己批評の機構**でした。「上位は非公開のFTや秘伝が必要」と思い込んでいましたが、**46点の公開ノートが手本として存在していた**のです（votesが低くて目立たなかっただけ）。しかも唯一のファインチューニング系ノートは43止まりで、「FTより素のgpt-oss-120b + 検証」という上位陣Write-upの結論とも一致していました。

私の42.5は、この公開46には3.5点届きませんでしたが、「次に何をやればいいか（critique-rlm/agenticを写経して検証機構を移植する）」が完全に明確になったのは大きな収穫でした。

### コンペの結末 — 大規模シェイクアップ

実はこのコンペ、**Private リーダーボードで大規模なシェイクアップ**が起きました（22,980登録 / 4,065参加者 / 3,450チームの規模）。主催者が公式に原因を説明しています。

- 強力な公開ノート（gpt-oss-120b 最適化版）が過去コンペ以上に影響力を持ち、**4,000以上のチームが同じモデルに集中**した
- gpt-oss-120b はファインチューニングが難しく、多くが推論時の工夫に流れた
- 結果が同じモデルの周りに密集し、**順位が「モデル固有のブレ（＝運）」に大きく左右される**ようになった。Privateを2回採点しても消しきれなかった

象徴的なのが、**Public 46 点だったチームが Private で 41.5 に転落**したり、**金メダル上位20チームのうち19チームが最後の1ヶ月は1度も提出していなかった**（たまたま早期に提出して放置した人が上位に残った）という事実です。主催者自身も「運の要素が大きかった」「将来のベンチマークは進歩を測るだけでなく前進させる設計が必要」と振り返っています。

### Privateで沈まなかった理由

前回のコンペでPublic 80位 → Private 467位という大転落を経験したので、今回 **Public と Private が完全一致（42.5 = 42.5）** したのは、個人的にはかなり嬉しい結果でした。とくに上記の通り**多くのチームが大きく順位を動かした中で自分はブレなかった**ので、運ゲー化した今回の環境では、これは数少ない「再現性のある成果」だったと思います。

当初は「temperature=1.0 という高温設定だと、2回採点で片方しか正解しない（0.5点）リスクが高いのでは」と心配していました。
しかし実際には全くブレませんでした。これは **8試行の多数決 + エントロピー重み付き投票 + Early stop の組合せが、個々の試行のブレを吸収して、最終回答自体を再現性のあるものにしてくれた** からだと考えています。

後から読んだ Nitarach 論文でも、**温度 T=1.0 で十分にエラーが脱相関する**（試行間のエラー相関がほぼゼロ）と独立に確認されており、「Private用に温度を下げるべき」という当初の懸念は杞憂だったことが裏付けられました。

前回の「Privateで沈む」失敗を踏まえて、投票による安定化を効かせられたのは、一つの収穫だったかなと思います。

# 次にやりたいこと

上位入賞者のWrite-upを踏まえると、自分の 42.5 から **公開スコアの天井（46帯）** に届くための最短経路は、意外にも「重い武器（FT）」ではなく「軽い道具（プロンプト + 検証）」でした。

1. **体系的な誤答分析プロンプト**（3位の核心）: リファレンス問題でLLMを大量に試行させ、誤答パターンを分類してプロンプトに注意喚起を組み込む → +1〜3点
2. **Sharper Single Prompt**（2位の核心）: 5段階プロンプトを保持しつつ、誤答回避リストを明示する → +1〜2点
3. **Verifier（GenSelect簡易版）**: pass@20とmajority-voteのギャップを回収する → +2〜3点
4. **複数提出でσ=1.7のブレを吸収**: 同じ設定で複数回提出して良いランを選ぶ → +0.5〜1点

①〜③だけでも 46点圏に届く可能性があり、FTは「時間があれば」の十分条件、という整理になりました。

しかも前述の通り、**Public 46 の `critique-rlm-like` と Public 45 の `aimo-agentic-solver` という「検証機構入りの公開ノート」が手本として既に存在**しています。次に数学推論系のコンペがあれば（あるいは復習として）、まずはこの2つを写経実行して、自分の42.5ノートに「答えを批評して解き直すループ」や「ツール付き検証エージェント」を移植するところから始めたいと思います。

ただし、前述の大規模シェイクアップが示すように、**「公開スコアの天井を上げること」と「Private で上位に残ること」は別問題**です（Public 46 のチームも Private では 41.5 に落ちました）。なので①〜③で天井を上げつつ、④の「複数回提出して分散を測り、ブレに強い設定を選ぶ」をセットで回すこと——つまり「公開46を踏み台にしつつ、運に左右されにくい安定性も同時に磨く」のが、次に目指すべき方向だと整理できました。
